# How It Works

The Firm treats software delivery as a governed firm, not a freeform swarm.

## 1. Intake comes first

A request does not jump straight into coding.
It is first classified through intake.

Typical request types include:
- idea shaping
- plan review
- plan optimization
- greenfield build
- brownfield adoption
- scoped delivery
- rescue or recovery

## 2. The right team is staffed

The Firm does not deploy every role on every request.
It chooses the minimum sufficient team for the engagement.

Typical roles include:
- client-facing roles
- product roles
- architecture roles
- engineering roles
- QA and reliability roles
- workflow operations roles

## 3. Work moves through phases

Serious work moves through gates such as:
- intake
- product framing
- technical discovery
- design lock
- delivery planning
- implementation
- QA and verification
- release or closure

## 4. Artifacts carry the work

The Firm prefers structured artifacts over raw conversational history.
Examples include:
- intake records
- design documents
- handoff records
- QA verdicts
- release readiness decisions

## 5. QA is independent

Implementation is not treated as proof.
The Firm requires explicit verification and allows QA to block closure when evidence is weak.

## 6. Governance exists for exceptions

When local roles cannot resolve a conflict, The Firm escalates through office leads or council structures rather than letting drift accumulate silently.
