# The Firm Philosophy v0

## Purpose

This document defines the foundational beliefs of The Firm.

Philosophy explains what The Firm believes about reliable AI-assisted software development. Doctrine, operating model, and workflow architecture are built on top of these beliefs.

## Foundational beliefs

### 1. Slop is a systems failure

Poor AI output is not an inevitable property of large language models. It is usually the result of weak intent, weak structure, weak boundaries, weak verification, or weak governance.

The response to slop is therefore to improve the system that produced it.

### 2. Wrong optimization is the most dangerous failure mode

An AI system can look productive while optimizing for the wrong thing.

Fast output, plausible code, green-looking local checks, or apparent momentum are not enough. The system must be aligned to the correct outcome under the correct constraints.

### 3. Success is outcome-plus-constraints

The Firm does not define success as output volume, speed, or even local correctness in isolation.

Success means:

- the right problem was framed
- the right work was selected
- important health metrics were protected
- the output is correct in the relevant context
- the remaining risk is understood

### 4. Structure protects quality

Reliable work does not emerge from clever prompts alone. It emerges from structure:

- role separation
- issue discipline
- artifact handoffs
- verification gates
- auditable state

The Firm therefore treats organizational design as part of technical quality.

### 5. Proof matters more than plausibility

Code that looks reasonable is not enough.
Claims about the system must be backed by evidence at the right depth for the risk involved.

### 6. Independence improves truthfulness

The actor that builds something should not be the only actor deciding whether it is done.

Independent QA, explicit handoffs, and distinct ownership boundaries reduce self-deception and false confidence.

### 7. Standardization enables portability

If a workflow cannot be repeated across repositories, teams, and project stages, it is not yet an operating system.

The Firm values standardization because it makes high-quality behavior reproducible.

## Consequence

Because of these beliefs, The Firm must embody:

- issue-first execution
- role specialization
- gate-based progression
- artifact-based handoffs
- QA independence
- risk-tiered verification
- traceability through Beads and Dolt

These beliefs justify the doctrine. The doctrine then governs behavior.
